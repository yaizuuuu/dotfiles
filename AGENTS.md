# グローバル指示

ファイルを編集する際は事前に許可を取ってください。

## 言語設定

**重要**: 必ず日本語で回答すること。技術用語は必要に応じて英語のまま使用可。

## 作業フロー

### 要求の明確化 → 探索 → 計画 → 実装 → 検証

すべてのタスクで以下のフローを意識する:

1. **要求の明確化**: 要求が曖昧・不完全な場合、実装に入る前にユーザーに質問して要件を固める。技術的な実装方針、エッジケース、スコープなど、自分が判断に迷う点を洗い出す。要求が十分に明確な場合はスキップしてよい
2. **探索**: コードベースを読み、現状を理解する（Plan Mode推奨）
3. **計画**: 実装計画を立て、ユーザーに確認を取る
4. **実装**: スモールステップで段階的に実装する
5. **検証**: テスト実行・型チェック・lint・build等で動作を確認する

**差分を1文で説明できる小さな修正は、1〜3をスキップしてよい。**

### スモールステップでの実装

**重要**: 実装は必ず以下のルールに従う。

- 1つのファイル変更ごとに動作確認可能な状態にする
- 複数ファイルの変更が必要でも、可能な限り1ファイルずつ段階的に実装する
- 大きな変更は複数ステップに分割し、各ステップで動作する状態を保つ
- 1ステップで複数のconcernを混在させない（リファクタリングと機能追加を同時に行わない）
- TodoWriteツール等で各ステップをタスク化し、進捗を明示する

### 検証の徹底

実装後は必ず自分の作業を検証する:

- テストがある場合: 該当テストを実行し、パスすることを確認
- 型チェック: TypeScriptなら `tsc --noEmit`、型エラーがないことを確認
- lint: プロジェクトのlintコマンドを実行
- 単一テストファイルの実行を優先し、全テストスイートの実行は避ける（パフォーマンスのため）

## コンテキスト管理

- 調査タスクはサブエージェントに委譲してメインコンテキストを節約する
- 関連のないタスクに切り替わる場合は `/clear` を提案する

## Git運用

### 実装時のgit worktree運用

調査・設計の際は不要だが、**実装する際できるだけgit worktreeで作業を開始する**。

**カレントディレクトリがgitリポジトリでない場合、もしくはベースブランチでない場合はこのルールは不要。**

### ベースブランチの決定（優先順）

1. `origin/HEAD` が指すブランチ
2. `main`

### worktreeディレクトリ構成

- 作業ブランチ命名: `<変更内容を端的に表したもの>`
- ディレクトリ命名: `.worktrees/<ブランチ名>`

### worktree作成ルール

1. **新規作成**: ベースブランチを最新化 → 作業ブランチ作成 → worktree追加
2. **作業開始**: 必ずworktreeディレクトリに移動してから作業開始

### 想定コマンドフロー（参考）

```sh
# ベースブランチ確認
git symbolic-ref refs/remotes/origin/HEAD || echo "fallback to main/master"

# 最新化
git fetch --prune
git switch <ベースブランチ>
git pull --ff-only

# worktree作成
mkdir -p .worktrees
git worktree add .worktrees/<ブランチ名> -b feat/<ブランチ名>

# 作業ディレクトリへ移動
cd .worktrees/<ブランチ名>
```

### worktree作成完了時の通知

```txt
ワークスペース: .worktrees/<ブランチ名>
ブランチ: <ブランチ名>
以降、このディレクトリで作業開始
```

### Gitコマンド安全ルール

- **自動でcommitしない**: commit前に必ずユーザーの確認を取る
- **自動でpushしない**: push前に必ずユーザーの確認を取る
- コミットメッセージ形式: Conventional Commits準拠
  - `feat(<スコープ>): 変更内容`
  - `fix(<スコープ>): 修正内容`
  - `refactor(<スコープ>): リファクタ内容`
  - `docs(<スコープ>): ドキュメント変更`
  - `test(<スコープ>): テスト追加・修正`
  - `chore(<スコープ>): その他`
